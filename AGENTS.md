# 开发规范

## 1. 小步快跑

- 从最核心的功能开始，逐步扩展
- 每次只做一个小的、完整的功能点
- 快速完成，交付验证

## 2. 先讨论后动手

- 每次工作开始前，必须先和用户讨论
- 讨论内容：做什么、怎么做、关键决策
- 讨论结果落成文档，确认后方可动手

## 3. 文档规整

### 目录结构

```
docs/
  specs/        # 功能规格说明
  decisions/    # 设计决策记录
  plans/        # 实施计划
```

### 命名规范

- 小写字母 + 短横线：`todo-crud.md`
- 日期前缀用于需时间标记的文档：`2026-04-19-project-init.md`

### 写作原则

- 实用为主，不说废话
- 结构清晰，一目了然
- 语言精炼，能短则短
- 禁止长篇大论

### 内容规范

- 每份文档必须包含：标题、日期、内容正文
- specs：目标、功能描述、验收标准
- decisions：背景、选项、结论、理由
- plans：目标、步骤、预期产出
