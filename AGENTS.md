# 开发规范

## 1. 小步快跑

- 从最核心的功能开始，逐步扩展
- 每次只做一个小的、完整的功能点
- 快速完成，交付验证

## 2. 先讨论后动手

- 每次工作开始前，必须先和用户讨论
- 讨论内容：做什么、怎么做、关键决策
- 讨论结果落成文档，确认后方可动手

## 3. 文档规整

### 目录结构

```
docs/
  specs/        # 功能规格说明
  decisions/    # 设计决策记录
```

### 命名规范

- 每个文档都必须使用 `YYYYMMDDHHmm-xxx.md`
- 时间前缀后接小写字母 + 短横线，例如：`202604250225-todo-importance.md`

### 写作原则

- 实用为主，不说废话
- 结构清晰，一目了然
- 语言精炼，能短则短
- 禁止长篇大论

### 内容规范

- 每份文档必须包含：标题、日期、内容正文
- specs：目标、功能描述、验收标准；只写最终规格，不写实施过程
- decisions：背景、选项、结论、理由；只记录关键取舍、范围变化、结构调整
- 不再使用 plans 目录
- 实施过程中的关键取舍、范围调整、结构变化，统一记录到 decisions
- 已经能在 specs 清楚表达的最终行为，不要在 decisions 里重复
